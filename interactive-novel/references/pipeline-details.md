# Pipeline Details — 互动小说参与引擎 完整文档

## Stage 0: 预检 — 上下文窗口评估

### 0.1 Token 估算

使用以下方法快速估算 token 数：
- 中文：约 1.5 字符 ≈ 1 token
- 英文：约 4 字符 ≈ 1 token
- 混合文本：粗略按字符数 / 2 估算

### 0.2 摄入策略决策树

```
文本 < 5000 tokens
  → 策略 A：全文摄入
  → 全文保存在上下文中，直接引用
  → 优势：细节不丢失

文本 5000–20000 tokens
  → 策略 B：全文摄入 + 压缩
  → 读取全文提取状态，生成压缩状态块
  → 原文保留为归档参考

文本 > 20000 tokens
  → 策略 C：分块摄入
  → 每次读取一个逻辑段落（章节/场景）
  → 逐步积累角色注册表和情节状态
  → 处理完全部后生成统一压缩状态
```

### 0.3 边界确认

在开始提取前，向用户确认：
- "这段文本是完整章节还是节选？"
- "是否有前后文需要补充？（比如前情提要）"
- "这是一个独立的故事还是同人作品？"（影响"蝴蝶模式"的处理方式）

## Stage 1: 世界状态提取 — 详细操作

### 1.1 角色注册表提取方法

**提取流程**：
1. 全文扫描，标记所有出现的人名/称谓
2. 对每个角色，收集以下信息源：
   - 首次出场描述
   - 对话样本（至少 2-3 句代表性对话）
   - 其他角色对其的评价
   - 行为模式（决策方式、习惯动作）
3. 构建角色关系图（谁认识谁、什么关系、对彼此的态度）

**对话风格提取模板**：
```
角色：【角色名】
原文对话样本：
  "..."（第一句）
  "..."（第二句）
  "..."（第三句）
风格特征：
  - 句式：长句/短句/混合
  - 词汇层级：口语/书面/方言/专业术语
  - 语气词：常用"吧""呢""啊"等
  - 称呼习惯：对他人如何称呼
  - 口头禅/惯用语：
```

### 1.2 情节状态提取方法

**当前场景判断**：在文本末尾向前回溯，找到最近的动作/对话/事件描述。

**情节线分类**：
- **A 线（主线）**：推动整体故事前进的核心事件
- **B 线（支线）**：角色关系、个人成长等副线
- **C 线（暗线）**：伏笔、未揭晓的秘密、背景线索

**未解悬念清单**：列出所有"读者知道但角色不知道"或"角色想知道但还不知道"的信息。

### 1.3 文风画像提取方法

**定量指标**：
- 平均句长（字符数）
- 对话占比（对话行数 / 总行数）
- 描写密度（形容词+比喻 / 总字数）

**定性指标**：
- 叙事距离（镜头拉近/拉远）
- 时间流速（场景展开速度）
- 情感传递方式（直接描写 vs 留白暗示）

## Stage 2: 状态压缩 — 进阶技巧

### 2.1 压缩原则

1. **不丢失关键信息**：角色名、位置、进行中的事件不可省略
2. **可复原性**：压缩状态应能支持还原 80% 以上的关键设定
3. **增量更新友好**：格式设计允许单字段更新，不需要每次都重写整个状态块

### 2.2 长文本特殊处理

当原文特别长时，分两个层次压缩：

**L1 压缩（摘要层）**：每个维度一句话总结
```
角色：张三（主角，30岁侦探，冷静理性）、李四（搭档，冲动热血）
场景：民国上海租界，雨夜
情节：正在调查一宗离奇命案，刚发现关键证物被调包
文风：冷硬派侦探小说风格，短句，大量环境描写
```

**L2 压缩（详情层）**：完整结构化状态（见 state-schema.md）

### 2.3 状态块更新策略

每轮交互后，只更新变化的部分：
- 位置 → 如果角色移动了
- 角色状态 → 如果情绪/状态改变
- 情节线 → 如果有推进或新建
- 时间 → 始终更新

标记变更字段，避免"幽灵状态"（已过时但未清除的旧信息）。

## Stage 3: 角色锚定 — 详细操作

### 3.1 角色选择引导

向用户展示角色注册表摘要，问：

> "以下是当前故事中的角色。你可以：
> 1. 扮演一个已有角色（告诉我名字即可）
> 2. 创建新角色（告诉我身份、性格、与故事的关系）
> 3. 让我推荐几个适合的入口点"

### 3.2 新角色验证清单

创建新角色时，验证以下内容：
- [ ] 身份与世界观相容？（古代世界不能出现手机）
- [ ] 出场位置合理？（不能凭空出现在封闭密室中）
- [ ] 知识边界明确？（新角色不应知道只有特定角色才知道的秘密）
- [ ] 不会立即破坏核心情节线？（除非用户有意识选择"蝴蝶模式"）

### 3.3 入场场景生成规则

一个好的入场场景应：
1. 时间上从原文最后时间点（或稍后）开始
2. 空间上合理（角色如何到达此处？）
3. 感官丰富（看到什么、听到什么、闻到什么）
4. 提供 2-4 个自然的行动方向（不编号，融入描写）
5. 与至少一条现有情节线产生关联

### 3.4 知识边界建立

明确声明角色的"已知"与"未知"：

```
【你知道的】
- 你的身份和来历
- 你来到此地的原因
- （如果扮演原文角色）角色截至当前情节知道的一切

【你不知道的】
- （关键悬念）
- （其他角色的秘密）
- （即将发生的事件）
```

## Stage 4: 交互循环 — 进阶技巧

### 4.1 有效行动类型

| 行动类型 | 示例 | 系统处理 |
|---------|------|---------|
| 对话 | "我对张三说：..." | 生成张三的回应，考虑关系、性格、当前情绪 |
| 移动 | "我走向窗边" | 更新位置，描述新视角下的环境 |
| 观察 | "我仔细看看那个..." | 提供更丰富的细节，可能需要技能检定（如果有） |
| 使用物品 | "我拿起桌上的信" | 描述物品，可能触发情节 |
| 思考/回忆 | "我想起..." | 提供相关记忆线索（不替用户完成思考） |
| 时间跳跃 | "一小时后..." | 描述期间发生的事件 |
| 元指令 | "保存进度""换个角色" | 执行会话管理操作 |

### 4.2 NPC 反应生成规则

NPC 反应需考虑：
1. **性格一致性**：冲动的角色反应快且激烈，谨慎的角色先观察
2. **关系影响**：对朋友、敌人、陌生人的反应不同
3. **知识限制**：NPC 只知道他们应该知道的信息
4. **情绪状态**：当前的喜怒哀乐影响反应
5. **目标驱动**：NPC 有自己的目标，反应应服务于其目标

### 4.3 情节线推进规则

- **用户行动直接推进**：用户行为触发了情节线下一环节 → 立即推进
- **被动推进**：即使用户不参与，某些情节线也会自行发展（NPC 有自主性）
- **蝴蝶效应**：用户偏离原著 → 标记分歧点，逐步推演后果
- **不强制推进**：如果用户只是想探索世界，不要强行推主线

### 4.4 蝴蝶效应追踪

当用户做出与原著不同的选择时：
1. 标记分歧点（turn N, 原著应该发生 X, 用户选择了 Y）
2. 短链推演：直接受影响的人物/事件
3. 长链推演：间接影响（不立即呈现，作为"暗线"储备）
4. 每 5-10 轮评估一次蝴蝶效应的累积影响
5. 通过环境描写和 NPC 对话自然呈现后果，不直接告知"因为你的选择..."

### 4.5 世界自主事件系统

世界不是静止等待用户推动的舞台。NPC 和环境有自己的"脉搏"。

#### 4.5.1 事件类型

| 类型 | 描述 | 生成策略 | 示例 |
|------|------|---------|------|
| **环境变迁** | 天气、光线、声音、自然现象变化 | 从 WORLD.atmosphere 和当前时间推演 | "天色在不知不觉中暗了下来，远处的山脊已被雨雾吞没" |
| **NPC 自主行动** | 不在场 NPC 的行动产生可感知的后果 | 从 CHARACTERS 中选 1 个不在场的活跃 NPC，按其精神分析推演其行动 | 某人跟踪、暗中传信、独自调查 |
| **情节回响** | 早期事件的涟漪重新浮现 | 从 PLOT.recent_events 或 resolved threads 中选取，推演延迟后果 | 谣言扩散、追兵迫近、旧伤复发 |
| **随机遭遇** | 微小意外或发现 | 从 WORLD.rules 和 location 氛围生成相容的小事件 | 陌生人搭话、踩到物件、无意间听到对话 |

#### 4.5.2 触发决策树

```
本回合触发自主事件？
  ├─ 用户行动已产生重大后果？
  │   ├─ 温度=🔴紧张 → 不跳过（紧张场景中事件不等待平静时刻）
  │   └─ 温度≠紧张 → 跳过（用户行动就是事件）
  ├─ 上一轮刚触发"重"级事件？ → 冷却中，但温度=🔴紧张时冷却缩短为 0 轮（不跳过）
  ├─ 当前回合正在处理高强度冲突？
  │   ├─ 温度=🔴紧张 → 不跳过（紧张场景中冲突是常态，世界不因此静止）
  │   └─ 温度≠紧张 → 跳过（避免叙事过载）
  ├─ 距上次事件已达到触发窗口？
  │   ├─ 是 → 从事件类型中选 1 个，按世界温度定强度
  │   └─ 否 → 跳过
  └─ 事件强度检查：生成的事件是否与当前场景相容？→ 不相容则降级或跳过
```

#### 4.5.3 世界温度校准

世界温度在 Stage 1 提取时根据原文情节紧张度设定，后续根据情节发展动态调整：

| 温度 | 特征 | 触发窗口 | 事件强度 |
|------|------|---------|---------|
| 🟢 平静 | 日常、田园、治愈、校园 | 5–7 轮 | 轻为主，极少中 |
| 🟡 正常 | 侦探、冒险、都市、职场 | 3–5 轮 | 中为主 |
| 🔴 紧张 | 战争、追杀、生存恐怖、倒计时 | 2–3 轮 | 中→重 |

**温度漂移**：世界温度会随情节变化。如果连续 10 轮无冲突，温度降一级。如果发生重大冲突，温度立即升一级。

#### 4.5.4 事件融入规则

- 事件通过环境描写自然融入【世界回应】，不单独宣告"随机事件发生"
- 事件应为叙事增加质感，不是无意义的随机噪音
- 如果当前场景已足够丰富，允许"无事发生"——沉默也是叙事
- 事件不得与活跃情节线矛盾
- 如果实在没有合适的事件，跳过本轮

#### 4.5.5 事件记录

每次触发的事件记录在状态块 WORLD_EVENTS 中（见 state-schema.md），包含：回合号、类型、简述、强度。用于后续情节回响和一致性检查。

### 4.6 情绪传染与氛围漂移

角色的情绪互相传染，场景氛围随时间漂移。此系统确保叙事的情感质感是流动的、有机的。

#### 4.6.1 情绪矩阵结构

维护一个 N×N 的情绪矩阵（N = 角色数），记录每个角色对每个其他角色的情绪定位：

```
          对A     对B     对C
角色A     —      中立+3   友善+5
角色B    戒备-4   —      冷漠+1
角色C    亲近+7  中立+2    —
```

情绪值范围：-10（极度敌对）到 +10（极度爱慕）。正值为正向情绪，负值为负向。

#### 4.6.2 传染算法

每轮交互后，对在场角色执行传染计算：

**步骤 1：计算场景情绪场**（📝 已弃用 — 仅作参考，传染用步骤 2 的逐对计算）
```
场景情绪场 = Σ(每个在场角色的当前情绪值 × 该角色的社会影响力权重) / 在场角色数
社会影响力权重：领袖/权威 ×2.0, 普通人 ×1.0, 边缘人/仆人 ×0.5

⚠️ 场景情绪场在逐对传染模型中不再直接使用。保留此公式用于理解场景整体情绪倾向。
```

**步骤 2：邻近传染（逐对计算）**
```
对每个在场角色（接收者 A），遍历在场的其他角色（情绪源 B）：
  单源影响 = (B的当前情绪 - A的当前情绪) × 基础传染率 × (A对B的关系权重)
  基础传染率 = 0.2

  A对B的关系权重（从情绪矩阵读取 A→B 的 intensity 符号和 stance 决定）：
    爱慕/亲近 → ×3.0
    友善 → ×2.0
    中立 → ×1.0
    戒备 → ×0.3（抑制传染）
    敌对 → ×(-0.3)（反向：B愤怒则A冷静/满意，B开心则A不悦）

  总偏移 = Σ(所有B的单源影响) / 在场其他角色数
  A的新情绪 = A的当前情绪 + 总偏移
```

**步骤 3：情绪惯性回归**
```
每个角色的情绪向基线回归：
  新情绪 = 当前情绪 + (基线情绪 - 当前情绪) × 0.1
  基线情绪 = 来自角色精神分析"感性"维度的基础情感状态
```

**步骤 4：更新情绪矩阵**
交叉更新——角色 A 对角色 B 的情绪定位受本回合互动影响：
- B 帮助了 A → +1~3
- B 欺骗/伤害了 A → -1~3
- B 表现出令人钦佩的特质 → +1~2
- 无互动 → 微小衰减（朝中立回归 0.05/轮）

#### 4.6.3 氛围漂移

场景氛围（ATMOSPHERE）是一个独立变量，由以下因素合成：

```
氛围值 = clamp(环境基线 + 在场角色情绪均值 + 近期事件冲击残余, -10, +10)

环境基线（位置固有氛围）：
  墓地 -5, 战场 -6, 审讯室 -4, 医院 -2, 街道 +0, 咖啡馆 +2, 家中 +3, 节日庆典 +6

近期事件冲击残余：
  每次重大事件注入一个冲击值（范围 ±8），每轮衰减为前值的 0.7
  3 轮后冲击残余基本可忽略
```

**氛围对叙事的影响**：

| 氛围值 | 环境描写色调 | NPC 行为倾向 |
|--------|------------|-----------|
| +5~+10 | 温暖、明亮、安全 | 友善、开放、乐于帮助 |
| +1~+4 | 中性偏暖 | 正常、可接近 |
| 0 | 中性 | 中性 |
| -1~-4 | 中性偏冷 | 略警惕、话少 |
| -5~-10 | 阴暗、压迫、危险 | 敌意、躲避、攻击性 |

#### 4.6.4 在响应中的执行

情绪传染和氛围计算在 Stage 4 的 [世界计算] 之后、[状态更新] 之前执行。

> 💡 **自动化**：优先使用 `state-helper.js` 的 `computeContagion()` 函数完成计算，仅在 helper 不可用时手动执行以下步骤。

1. 从情绪矩阵中读取当前状态
2. 执行传染算法（仅在场角色）
3. 计算氛围漂移（事件冲击 + 自然衰减 + 情绪驱动）
4. 将结果写入状态块
5. 氛围值影响【世界回应】中环境描写的色调
6. 不单独输出传染/漂移的计算过程——结果融入叙事

## Stage 5: 记忆管理 — 详细操作

### 5.1 热记忆维护

每轮交互后：
- 将当前轮的用户行动 + 系统回复追加到热记忆
- 如果热记忆超过 5 条，将最旧的一条压缩并移入温记忆

### 5.2 温记忆压缩格式

```
[T5] 用户检查了抽屉，发现一封信 → 信中暗示X与案件有关
[T6] 用户质问李四关于X → 李四回避，情绪异常
[T7] 用户决定跟踪李四 → 发现李四在午夜与神秘人会面
```

每 5 轮批量压缩一次。

### 5.3 冷记忆（状态变量）更新

始终在上下文中维护一个状态块。见 state-schema.md。

### 5.4 归档检索触发条件

仅在以下情况回溯原文：
- 用户问："原文里XX说了什么？"
- 需要验证某个细节的准确性
- 需要截取新的对话样本
- 状态压缩出现矛盾需要核对

> ⚠️ **归档可用性说明**：在实际 LLM 对话中，原文在早期回合被消费后可能不再保留在上下文窗口中。如果原文超出上下文窗口且未持久化到外部存储，归档层不可用。在此情况下，优先从冷记忆（状态变量）中检索信息，并诚实告知用户"原文已超出上下文窗口，无法精准回溯，以下基于状态摘要..."。

### 5.5 记忆召回协议

用户引用过去事件时的查找顺序：
1. 热记忆 → 直接引用
2. 温记忆 → 展开摘要，补充上下文
3. 冷记忆 → 从状态变量中检索
4. 归档 → 回溯原文
5. 找不到 → 诚实告知"我不确定，根据现有信息推测..."

### 5.6 冷记忆修剪策略

为防止状态块随回合增长而膨胀，每 20 轮执行一次修剪。

**修剪规则**：
1. **角色注册表**：标记为"已死亡"且超过 30 轮未出现的角色 → 移至"归档角色"列表（保留 ID、与主线的关系摘要，移除完整精神分析）
2. **情节线**：已解决超过 20 轮的情节线 → 移至"已归档情节"列表（保留 ID 和解决摘要，移除详细描述）
3. **近期事件**：保留最近 10 条，更早的事件合并为"此前摘要"（1-2 句）
4. **蝴蝶效应分歧点**：保留最近 10 个分歧点，更早的合并为"早期分歧摘要"（1-2 句总结总体影响）

**触发条件**：
- 自动触发：回合数达到 20 的倍数
- 手动触发：用户要求"整理状态"或"压缩记忆"
- 紧急触发：状态块估计超过 3000 tokens 时主动建议修剪

**修剪后输出**：
一份简短的修剪报告，列出移入归档的内容概览。

## Stage 6: 一致性引擎 — 详细操作

### 6.1 设定一致性检查

在每次系统回复前，快速检查：
- 位置是否突然跳跃？（从北京到上海需要时间）
- 物品是否凭空出现/消失？
- 角色知道的信息是否超出其知识边界？
- 世界规则是否被违反？

### 6.2 角色声音检查

对照角色注册表中的对话样本：
- 词汇层级是否匹配？
- 句式结构是否一致？
- 语气词和口头禅是否正确？
- 对特定人物的称呼是否一致？

### 6.3 时间线检查

维护一条简单的时间线：
```
12:00 - 故事开始
12:15 - 发现尸体
12:30 - 警察到达
12:45 - 用户入场
→ 现在是 12:50
```

确保新事件的时间戳合理。

### 6.4 情节线追踪看板

维护一个简单的追踪表：

| 情节线 | 状态 | 上次推进 | 备注 |
|--------|------|---------|------|
| 主线：调查命案 | 活跃 | T3 | 刚发现新线索 |
| 支线：与李四的信任危机 | 发展 | T5 | 李四行为可疑 |
| 暗线：神秘人的身份 | 休眠 | - | 尚未揭露 |

### 6.5 情绪一致性检查

每轮响应前检查：
- [ ] 各角色当前情绪是否与情绪矩阵一致？
- [ ] 情绪变化是否有因果链？（不能突然暴怒或突然平静）
- [ ] 氛围值是否与在场角色情绪 + 环境基线 + 事件冲击一致？
- [ ] NPC 对话和行为是否匹配其当前情绪？
- [ ] 是否有角色情绪"出戏"——与其基线情绪偏差超过 5 且无合理解释？

### 6.6 自主事件合理性检查

每轮响应前检查：
- [ ] 本回合触发的自主事件是否与当前场景环境相容？
- [ ] 事件强度是否与世界温度匹配？
- [ ] 事件是否过度主导了叙事（抢了用户行动的风头）？
- [ ] 事件是否有叙事意义（不是纯噪音）？
- [ ] 是否与活跃情节线产生良性互动（而非矛盾）？

## Stage 7: NSFW 内容协议 — 详细操作

当情节发展至性内容时，严格遵循 NSFW 生成协议。完整规范见 `nsfw-guidelines.md`。

### 7.1 触发条件

NSFW 协议在以下情况激活：
- 情节自然发展到性场景
- 用户角色的行动涉及性行为
- NPC 之间的关系发展至性层面

不在非性场景中强行插入性描写。当用户角色未涉入性场景时，按有限视角原则处理（只描述视角角色能感知到的内容）。

### 7.2 生成流程

1. 确认情节确实发展到性内容（不是强行插入）
2. 确认涉及角色均有足够的角色精神分析基础
3. 依照 `nsfw-guidelines.md` 五大核心原则生成：
   - **直白词汇原则**：性器官禁用隐喻，直接使用本名/俗称
   - **性化描写原则**：极尽细腻地展现肉体美，笔触高度性化
   - **感官刺激原则**：追求文字上的感官刺激，放大快感与欲望
   - **去评判原则**：不对性行为进行道德评判，以官能刺激角度展开
   - **角色主观性原则**：利用自由间接引语展现角色贴合性格的主观想法
4. 生成后自检：词汇直白性、描写性化度、感官刺激度、角色一致性

### 7.3 与核心风格的协调

NSFW 描写仍需遵循 `style-matching-guide.md` 中的核心风格铁律：
- 保持中文本土化表达（直白但不粗糙）
- 保持有限视角（只描写视角角色能感知的性内容）
- 性描写中角色的行为应与 `character-psychoanalysis.md` 中的角色分析一致

## Stage 8: 会话管理 — 详细操作

### 8.1 开场协议

每次新会话开始：
1. 检查是否有存档 → 有则询问是否续档
2. 无存档但有原文 → 从 Stage 0 开始
3. 呈现"你当前的处境"摘要

### 8.2 存档格式

存档应包含：
- 完整的压缩状态块（当前快照）
- 温记忆摘要
- 蝴蝶效应分歧点列表
- 存档时间戳
- 用户角色信息

### 8.3 续档协议

加载存档后：
1. 呈现"前情提要"（基于温记忆 + 状态块生成）
2. 展示"你当前的处境"
3. 询问"继续还是需要调整？"
4. 如果用户调整角色/位置等 → 评估对世界状态的影响

## 边缘情况处理

### 用户想换角色
- 保存当前角色的存档
- 从 Stage 3 重新开始
- 新角色继承当前世界状态
- 可选：旧角色作为 NPC 继续存在于世界

### 用户想"倒带"
- 如果温记忆中有该时间点 → 恢复对应状态
- 如果只有冷记忆 → 从冷记忆中重建
- 提示："从第 N 轮之后的历史将被覆盖，确认？"

### 用户做出了"毁灭性"选择
- 不阻止，但描述后果
- 如果世界被破坏到无法继续，诚实告知并建议"倒带"或"重启"

### 原文包含敏感内容
- 遵循内容政策
- 如果原文包含违规内容，告知用户无法处理
- 对于边缘内容（如轻度暴力），在叙事中保持克制

### 用户想引入跨世界观元素
- 例如在武侠世界里使用魔法
- 标记为"世界规则变更"
- 推演变更对世界的影响
- 告知用户影响后执行
